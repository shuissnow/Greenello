## 1. パスワードハッシュ化

- **脅威:** DB 漏洩時の平文パスワード露出
- **対策方針:** bcrypt でハッシュ化して保存。平文は一切保存しない
- **実装詳細:**
  - アルゴリズム: bcrypt
  - コストファクター: 12（ハッシュ生成に約250ms程度）
  - 使用ライブラリ: `BCrypt.Net-Next`
  - パスワード変更時は新ハッシュで上書き。旧ハッシュは保持しない
  - bcrypt の入力長制限（72バイト）に合わせてパスワード最大長を72文字に設定
- **対象ファイル/レイヤー:** `AuthService.cs`

## 2. サーバーサイドセッション管理

- **脅威:** JWT 漏洩時の長期間悪用、セッション無効化不可
- **対策方針:** ASP.NET Core のサーバーサイドセッションを使用。セッションIDのみ Cookie に保持
- **実装詳細:**
  - セッション保存先: インメモリ（初期実装）
  - Cookie 名: `greenello_session`
  - Cookie 属性: `HttpOnly`（JS からアクセス不可）/ `Secure`（本番のみ）/ `SameSite=Strict`
  - アイドルタイムアウト: 30分（操作のたびに延長）
  - 絶対タイムアウト: 7日間
  - ログアウト時・パスワード変更時にセッションを即時無効化
- **対象ファイル/レイヤー:** `Program.cs`（Session 設定）, `AuthController.cs`

## 3. SameSite=Strict Cookie による CSRF 対策

- **脅威:** 悪意のあるサイトからの偽リクエストによるユーザー操作の偽造
- **対策方針:** `SameSite=Strict` により、クロスサイトリクエストには Cookie が送信されない
- **実装詳細:**
  - Cookie の SameSite 属性を `Strict` に設定
  - CSRF Token 方式は採用しない（学習目的・SameSite=Strict で十分と判断）
  - API は JSON リクエストのみ受け付ける（form POST は不使用）
- **対象ファイル/レイヤー:** `Program.cs`（Cookie Policy 設定）

## 4. アカウントロック（ブルートフォース対策）

- **脅威:** パスワードの総当たり攻撃
- **対策方針:** 連続ログイン失敗10回でアカウントを30分ロック
- **実装詳細:**
  - users テーブルの `failed_login_count` と `locked_until` で管理
  - ロック閾値: 10回連続失敗
  - ロック期間: 30分（`locked_until = NOW() + 30分`）
  - 成功ログイン時: `failed_login_count = 0`, `locked_until = NULL` にリセット
  - エラーメッセージでメールアドレス存在の有無を明示しない（`INVALID_CREDENTIALS` で統一）
- **対象ファイル/レイヤー:** `AuthService.cs`, `UserRepository.cs`

## 5. SQL インジェクション対策

- **脅威:** 悪意のある SQL 文の埋め込みによる不正データ操作
- **対策方針:** EF Core の LINQ クエリとパラメータ化クエリを使用。生 SQL を使用しない
- **実装詳細:**
  - 全てのクエリを EF Core LINQ 式で記述する（`context.Users.Where(u => u.Email == email)` など）
  - 動的フィルター条件も `IQueryable` を連結して構築し、EF Core にパラメータ化を委譲する
  - 生 SQL（`FromSqlRaw`）は原則不使用。やむを得ず使用する場合はパラメータを必ず使用する
- **対象ファイル/レイヤー:** 全 Repository クラス

## 6. XSS 対策

- **脅威:** 悪意のあるスクリプト埋め込みによるクッキー盗取・画面改ざん
- **対策方針:** React の JSX が自動的にユーザー入力をエスケープする仕組みを活用
- **実装詳細:**
  - JSX の `{}` 式は React が自動でエスケープするため、`dangerouslySetInnerHTML` を使用しない
  - ユーザー入力を HTML として解釈する機能は実装しない
  - Cookie は `HttpOnly` 属性付きのため JS からアクセス不可
- **対象ファイル/レイヤー:** 全フロントエンドコンポーネント

---

## 7. CORS 設定

- **脅威:** 不正なオリジンからの API アクセス
- **対策方針:** フロントエンドのオリジンのみ許可する CORS ポリシーを設定
- **実装詳細:**
  - 許可オリジン: 環境変数 `FRONTEND_URL` で指定（ローカル: `http://localhost:5173`、本番: `https://yourdomain.com`）
  - 許可メソッド: GET, POST, PATCH, DELETE
  - 許可ヘッダー: `Content-Type`
  - `AllowCredentials`: 有効（Cookie 送受信のため）
  - ワイルドカード（`*`）は使用しない
- **対象ファイル/レイヤー:** `Program.cs`（CORS 設定）

## 8. エラーレスポンスのスタックトレース非公開

- **脅威:** 内部構造・使用技術・DB スキーマの漏洩
- **対策方針:** 全ての例外をミドルウェアでキャッチし、統一エラー形式のみ返す
- **実装詳細:**
  - `ErrorHandlingMiddleware` で未ハンドリング例外を全てキャッチ
  - レスポンスに含めるのは `error.code` と `error.message` のみ
  - スタックトレース・例外クラス名・DB エラー詳細はサーバーログにのみ記録
  - `ASPNETCORE_ENVIRONMENT=Production` 時は ASP.NET Core の developer exception page を無効化
- **対象ファイル/レイヤー:** `Middleware/ErrorHandlingMiddleware.cs`, `Program.cs`

---

## 9. HTTPS 強制（本番環境）

- **脅威:** 通信の盗聴・Cookie の平文送信
- **対策方針:** 本番環境では Nginx で HTTPS を強制し、HTTP → HTTPS リダイレクトを設定
- **実装詳細:**
  - AWS ACM で SSL/TLS 証明書を取得
  - Nginx で HTTPS（443）を終端し、バックエンドには HTTP で転送
  - HTTP（80）へのアクセスは 301 で HTTPS にリダイレクト
  - Cookie の `Secure` 属性は `ASPNETCORE_ENVIRONMENT=Production` 時のみ有効
- **対象ファイル/レイヤー:** `nginx.conf`

## 10. 認可チェック順序の遵守

- **脅威:** チェック順序の誤りによる権限昇格・情報漏洩
- **対策方針:** 全 API リクエストで以下の順序で認可チェックを実施する
- **実装詳細:**
  1. 認証チェック（セッション有効か）→ 無効なら 401
  2. ボードアクセスチェック（boards.owner_user_id と一致するか）→ 非オーナーなら 403
  3. リソース存在チェック（対象 ID が存在するか）→ なければ 404

  > 403 より先に 404 を返すと、リソース存在有無が漏洩するため順序を遵守する
- **対象ファイル/レイヤー:** 全 Controller / Service
