# 詳細設計フェーズ - 作業ルール

## 1. インプット

- `docs/02-basic-design/` の確定済み基本設計書を参照してください

## 2. アウトプットの粒度

- `詳細設計成果物一覧.md` に記載された成果物に従って出力してください
- 1ファイルが肥大化し関心事が複数になった場合は、フォルダを作成して適切な粒度でファイルを分割してください
- 「この設計書を読めば実装できる」物理レベルで記述してください

### DB 設計の粒度
- 物理名・型・制約・インデックスまで定義してください

### API 設計の粒度
- 全フィールドの JSON スキーマ・バリデーションルール・エラーコード・HTTP ステータスコードまで定義してください

### クラス・モジュール設計の粒度
- 責務・引数・戻り値まで定義してください

### ビジネスロジックの粒度
- 処理フロー・分岐条件・例外ハンドリングを擬似コードまたはフローチャートで記述してください
- 実際のコードは書かないでください（実装はコーディングフェーズで行います）

## 3. 書いてはいけないこと

- 実際のコードは書かないでください
- 基本設計で決まっていない方針を新規に設計しないでください（基本設計に戻って確認してください）
- 要件定義・基本設計の内容を転記・繰り返さないでください

## 4. Claude への判断指示

- 基本設計で `[検討中]` / `[TBD]` の項目に依存する実装詳細が必要な場合は、書かずにユーザーに確認してください

## 成果物一覧

### ロジック・処理設計

| 成果物 | 区分 | 作成が必要な理由 |
|-------|------|----------------|
| バックエンド・クラス・モジュール設計書 | 主要文書 | 各クラス・モジュールの責務・依存関係・インターフェースを定義します。実装者が迷わず開発できる設計の拠り所となります。 |
| フロントエンド・モジュール設計書 | 主要文書 | コンポーネント構成・状態管理・画面とAPIの接続方法を定義します。フロントエンド開発の実装指針となります。 |
| ビジネスロジック設計書 | 実装仕様 | 処理の分岐・繰り返し・例外ハンドリングを擬似コードやフローチャートで記述します。コーディング前に処理の抜け漏れや論理矛盾を発見でき、手戻りコストを削減できます。 |

### データ設計

| 成果物 | 区分 | 作成が必要な理由 |
|-------|------|----------------|
| テーブル定義書 | DB物理設計 | 論理ER図を実際の DBMS に合わせて物理化します。物理名・型・制約・インデックス・外部キーを確定することで、パフォーマンス問題やデータ不整合を実装前に防ぎます。 |

### インターフェース・連携設計

| 成果物 | 区分 | 作成が必要な理由 |
|-------|------|----------------|
| API 詳細設計書 | 連携仕様 | エンドポイント・HTTP メソッド・Request/Response の完全な JSON スキーマ・バリデーションルール・エラーコードを定義します。フロントエンドとバックエンドの実装齟齬を防ぐ契約書となります。 |

### インフラ・運用設計

| 成果物 | 区分 | 作成が必要な理由 |
|-------|------|----------------|
| CI/CD 設計書 | 自動化設計 | ビルド・テスト・デプロイのパイプラインを定義します。手動デプロイのミスを防ぎ、リリースの品質と速度を安定させます。 |
| ログ設計書 | 運用設計 | ログの出力形式・レベル・保持期間を定義します。障害発生時の原因調査と運用監視の基盤となります。 |
| 環境変数定義書 | 設定管理 | 環境ごとに異なる設定値を一覧化します。設定漏れ・環境差異によるバグを防ぎます。 |
| セキュリティ設計書 | セキュリティ | XSS 対策・入力バリデーション方針・TLS 終端方式などセキュリティ実装の詳細を定義します。 |

### テスト・品質

| 成果物 | 区分 | 作成が必要な理由 |
|-------|------|----------------|
| テスト設計書 | テスト設計 | 単体テストのテストケース・期待値・テスト手順を定義します。詳細設計と同時に作成することで設計レビューを兼ねられます。 |
