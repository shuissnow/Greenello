# ビジネスロジック設計書

## 1. ユーザー登録処理

- **概要:** メールアドレスとパスワードで新規ユーザーを登録する
- **トリガー:** POST /auth/register
- **処理フロー:**
  1. メールアドレス・パスワードのバリデーション（形式・文字数・文字種）
  2. users テーブルで同一メールアドレスの存在確認
  3. 重複があれば `CONFLICT` エラーを返す
  4. bcrypt（コストファクター: 12）でパスワードをハッシュ化
  5. users テーブルに INSERT（failed_login_count=0, locked_until=null）
  6. 作成したユーザーオブジェクト（パスワードハッシュ除く）を 201 で返す
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | メールアドレス重複 | 409 CONFLICT を返す |
  | バリデーション違反 | 422 VALIDATION_ERROR を返す |
- **例外・エラーケース:**
  | ケース | 対応 |
  |--------|------|
  | DB 接続エラー | 500 INTERNAL_ERROR・ERROR ログ記録 |
- **参照API / テーブル:** POST /auth/register, users

## 2. ログイン認証処理

- **概要:** メールアドレスとパスワードを検証し、セッションを開始する
- **トリガー:** POST /auth/login
- **処理フロー:**
  1. メールアドレスで users テーブルを検索
  2. ユーザーが存在しない場合は `INVALID_CREDENTIALS` を返す（メール存在有無を明示しない）
  3. `locked_until` が現在時刻より未来であれば `ACCOUNT_LOCKED` を返す
  4. bcrypt でパスワードを検証
  5. パスワード不一致の場合:
     - `failed_login_count + 1` を UPDATE
     - 新カウント < 10: `INVALID_CREDENTIALS` (401) を返す
     - 新カウント >= 10: `locked_until = NOW() + 30分` を UPDATE → `ACCOUNT_LOCKED` (423) を返す
  6. パスワード一致の場合:
     - `failed_login_count = 0`, `locked_until = null` を UPDATE
     - セッションにユーザーIDを書き込む
     - ユーザーオブジェクトを 200 で返す
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | ユーザーが存在しない | 401 INVALID_CREDENTIALS |
  | アカウントロック中 | 423 ACCOUNT_LOCKED |
  | パスワード不一致（9回以下） | failed_login_count++ → 401 INVALID_CREDENTIALS |
  | パスワード不一致（10回目） | locked_until 設定 → 423 ACCOUNT_LOCKED |
  | パスワード一致 | カウントリセット → セッション開始 → 200 |
- **例外・エラーケース:**
  | ケース | 対応 |
  |--------|------|
  | DB 接続エラー | 500 INTERNAL_ERROR・ERROR ログ記録 |
- **参照API / テーブル:** POST /auth/login, users

## 3. パスワード変更処理

- **概要:** 現在のパスワードを検証した上で新しいパスワードに変更し、全デバイスのセッションを一括無効化する
- **トリガー:** PATCH /auth/password
- **処理フロー:**
  1. セッションからユーザーIDを取得
  2. users テーブルから対象ユーザーを取得
  3. 現在のパスワードを bcrypt で検証
  4. 不一致の場合は `INVALID_CREDENTIALS` を返す
  5. 新パスワードのバリデーション（文字数・文字種）
  6. 新パスワードを bcrypt ハッシュ化
  7. users テーブルの `password_hash` を UPDATE
  8. 全デバイスの既存セッションを一括無効化する（現在のセッション含む他端末も強制ログアウト）
  9. 204 を返す（フロントエンドはログイン画面へリダイレクト）
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | 現在パスワード不一致 | 401 INVALID_CREDENTIALS |
  | 新パスワードのバリデーション違反 | 422 VALIDATION_ERROR |
- **参照API / テーブル:** PATCH /auth/password, users

## 4. タスク作成処理

- **概要:** ボードにタスクを作成する。初期ステータスは `inbox`
- **トリガー:** POST /boards/:boardId/tasks
- **処理フロー:**
  1. セッションからユーザーIDを取得
  2. ボードオーナーか確認（非オーナーなら 403）
  3. リクエストボディのバリデーション
  4. `assigneeUserId` が指定された場合、そのユーザーが存在するか確認
  5. tasks テーブルに INSERT（status=`inbox`, created_by_user_id=ログインユーザー）
  6. 作成したタスクオブジェクトを 201 で返す
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | ボード非オーナー | 403 FORBIDDEN |
  | バリデーション違反 | 422 VALIDATION_ERROR |
  | assigneeUserId のユーザーが存在しない | 422 VALIDATION_ERROR |
- **参照API / テーブル:** POST /boards/:boardId/tasks, tasks, users

## 5. サブタスク作成処理（2階層制限）

- **概要:** 既存タスクにサブタスクを追加する。親タスクがサブタスクの場合はエラー
- **トリガー:** POST /tasks/:taskId/subtasks
- **処理フロー:**
  1. セッションからユーザーIDを取得
  2. 親タスク（:taskId）を tasks テーブルから取得
  3. 親タスクが存在しない場合は 404
  4. 親タスクの `parent_task_id` が NULL でない場合（= 既にサブタスク）、422 VALIDATION_ERROR
  5. ボードオーナーか確認
  6. タイトルのバリデーション
  7. tasks テーブルに INSERT（parent_task_id = :taskId, status = `inbox`）
  8. 作成したサブタスクオブジェクトを 201 で返す
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | 親タスクが存在しない | 404 NOT_FOUND |
  | 親タスクが既にサブタスク | 422 VALIDATION_ERROR（2階層制限） |
  | ボード非オーナー | 403 FORBIDDEN |
- **参照API / テーブル:** POST /tasks/:taskId/subtasks, tasks

## 6. タスク削除処理（連鎖削除）

- **概要:** タスクを削除する。親タスクの場合はサブタスクを先に削除する
- **トリガー:** DELETE /tasks/:taskId
- **処理フロー:**
  1. セッションからユーザーIDを取得
  2. タスクを tasks テーブルから取得（存在しない場合 404）
  3. ボードオーナーか確認
  4. タスクの `parent_task_id` が NULL の場合（親タスク）、サブタスクを一括 DELETE
  5. タスク本体を DELETE
  6. 204 を返す
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | タスクが存在しない | 404 NOT_FOUND |
  | ボード非オーナー | 403 FORBIDDEN |
  | 親タスクの削除 | サブタスクを先に DELETE してから親を DELETE |
  | サブタスクの削除 | そのタスクのみ DELETE |
- **参照API / テーブル:** DELETE /tasks/:taskId, tasks

## 7. タスクステータス変更（Drag & Drop）

- **概要:** ドラッグ&ドロップによるカラム間移動でステータスを変更する。フロントはオプティミスティック更新を使用
- **トリガー:** カードのドロップ → PATCH /tasks/:taskId
- **処理フロー:**
  **フロントエンド:**
  1. ドロップ先カラムのステータスを確認
  2. TanStack Query キャッシュを即時更新（オプティミスティック）
  3. PATCH /tasks/:taskId を送信 `{ "status": "<new_status>" }`
  4. 成功: キャッシュを invalidate して最新データを再フェッチ
  5. 失敗: キャッシュを snapshot に復元（ロールバック）・エラートーストを表示

  **バックエンド:**
  1. セッション認証・ボードオーナー確認
  2. tasks テーブルの status と updated_at を UPDATE
  3. 更新後のタスクオブジェクトを 200 で返す
- **参照API / テーブル:** PATCH /tasks/:taskId, tasks

## 8. 期限切れタスク通知ジョブ

- **概要:** 毎日 08:00 JST に期限超過の未完了タスクを抽出し、作成者・担当者にメールで通知する
- **トリガー:** NotificationHostedService による定期実行
- **処理フロー:**
  1. Hosted Service が 08:00 JST に起動
  2. 以下の条件でタスクを SELECT:
     - `status IN ('inbox', 'in_progress')`
     - `due_date < CURRENT_DATE (JST)`
     - `due_date >= CURRENT_DATE - 30日`
  3. 各タスクについて通知対象ユーザーを特定（作成者・担当者）
  4. 作成者と担当者が同一ユーザーの場合は1通のみ
  5. notification_logs で当日の送信済みレコードを確認（重複防止）
  6. 未送信のみ SendGrid API でメール送信
  7. 送信成功後に notification_logs に INSERT
  8. 送信失敗時: 最大3回リトライ（5分後・30分後・1時間後）
  9. 3回失敗後は ERROR ログに記録（再送しない）
  10. 1件の失敗が他のタスクの処理に影響しないよう、タスク単位で try-catch
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | 当日送信済みレコードあり | スキップ（重複送信防止） |
  | 作成者と担当者が同一 | 1通のみ送信 |
  | SendGrid API エラー | リトライ（最大3回） |
  | リトライ上限到達 | ERROR ログ記録 |
- **参照API / テーブル:** tasks, notification_logs, SendGrid API

## 10. CSV エクスポート処理

- **概要:** フィルター条件に合うタスクを UTF-8 BOM 付き CSV として同期で返す
- **トリガー:** GET /boards/:boardId/tasks/export
- **処理フロー:**
  1. セッション認証・ボードオーナー確認
  2. クエリパラメータからフィルター条件を取得
  3. フィルター条件で tasks テーブルを SELECT（サブタスク含む）
  4. 取得件数が 0 件の場合は 400 `EXPORT_NO_TASKS` を返す
  5. メモリ上でレコードを CSV 文字列に変換（ヘッダー行 + データ行）
  6. ステータス等は日本語ラベルに変換（例: `inbox` → `Inbox`）
  7. UTF-8 BOM（`\xEF\xBB\xBF`）を先頭に付与
  8. `Content-Type: text/csv; charset=utf-8` でレスポンスを返す
- **条件分岐:**
  | 条件 | 処理 |
  |------|------|
  | フィルター結果が 0 件 | 400 EXPORT_NO_TASKS |
  | DB エラー | 500 INTERNAL_ERROR |
- **参照API / テーブル:** GET /boards/:boardId/tasks/export, tasks, users
