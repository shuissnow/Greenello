# プロダクト要求定義書

**アプリケーション名**: Greenello

## 背景・目的

### 背景

Web アプリケーション開発の技術的学習として、Trello 風のタスク管理アプリ「Greenello」を開発する。Kanban 形式を題材に選んだ理由は、CRUD・認証・複数ユーザー共有・非同期処理など実務で求められる機能を一つのアプリで網羅しやすいためである。

### 目的

以下のスキルを実践的に習得することを目的とする。

- セッションベース認証の設計・実装（ログイン・ログアウト・パスワード変更・セッション管理）
- REST API 設計と C# .NET Web API による実装
- React + Vite を用いた SPA フロントエンド開発（状態管理・Drag & Drop）
- PostgreSQL によるリレーショナル DB 設計と操作
- レイヤー分離（Presentation / Application / Domain / Infrastructure）を意識した設計と、各レイヤーの責務を説明できる状態の達成

**達成基準:**

以下をすべて満たした時点でプロジェクト完了とする。

- 認証フロー（新規登録・ログイン・ログアウト・パスワード変更）が正常に動作すること
- タスクの CRUD（作成・閲覧・編集・削除）が動作すること
- 4表示形式がそれぞれ動作すること（カンバン: タスクカードの表示と Drag & Drop によるステータス変更が行えること。リスト: タスク一覧のソート・フィルターが機能すること。カレンダー: 期限日にタスクが表示されること。進捗サマリー: ステータス別件数の棒グラフが表示されること）
- CSV エクスポートおよびメール通知（期限切れ通知）が動作すること
- 上記機能について、手動テストチェックリスト（`docs/07-system-test/` で定義予定）の全項目が Pass であること
- AWS へのデプロイが完了し、ログインからタスク作成までの一連操作が外部ネットワークから行えること

## プロダクトゴール

### 機能ゴール

- 複数ユーザーが共同でタスクを管理できる Kanban ベースの Web アプリを構築する
- ログイン後は自身が所属するボード一覧をホーム画面に表示し、ボードを選択してボード詳細画面へ遷移する
- タスクはサブタスク（2 階層まで）を持てる（詳細は `docs/01-requirements/スコープ定義書.md` 参照）
- 以下の4形式でタスクを可視化する
  - **カンバン**: ステータス別カラムにタスクカードを表示。Drag & Drop で移動可能
  - **リスト**: タスクを一覧テーブルで表示。ソート・フィルター対応
  - **カレンダー**: 期限日をカレンダー上にタスクとして表示
  - **進捗サマリー**: ステータス別（未着手 / 進行中 / 完了）のタスク件数を棒グラフで可視化（ステータスの詳細定義は `docs/01-requirements/スコープ定義書.md` 参照）

### 技術ゴール

- React + Vite（SPA）・C# .NET Web API・PostgreSQL・Docker・AWS を用いたフルスタック構成のアプリケーションを完成させる
  - EC2: Nginx で React の静的ビルドを配信し、C# .NET Web API をリバースプロキシで公開するアプリサーバー（CloudFront は学習スコープ外のため使用しない）
  - データベースホスティング: [TBD]（RDS または EC2 上のコンテナで後続フェーズに確定）
  - SendGrid: 期限切れ通知メールの送信（外部メール送信サービス）
  - CSV エクスポート: サーバーがメモリ上で CSV を生成し、クライアントへ直接レスポンスする（ファイルストレージは使用しない）

### 設計ゴール

- レイヤー分離（Presentation / Application / Domain / Infrastructure）を意識した設計を行い、機能追加時の影響範囲を局所化できる構成とする

## ターゲットユーザー

| ユーザー種別 | 説明 |
|-----------|------|
| 未認証ユーザー | アカウントを持たない訪問者。新規登録画面・ログイン画面のみアクセス可能。それ以外の全機能は認証必須 |
| 登録ユーザー | アカウントを持ち、タスクの作成・編集・削除・閲覧を行うメインユーザー |
| ボードオーナー | ボードを作成したユーザー。そのボードにアクセスできる唯一のユーザー |

### 権限マトリクス（ボード外操作）

アカウントレベルの操作（ボードを問わず適用）。

| 操作 | 未認証 | 登録ユーザー（全員） |
|-----|--------|-------------------|
| 新規登録・ログイン | ○ | —（既認証） |
| ボードの新規作成 | × | ○（作成者がそのボードのオーナーとなる） |
| アカウント設定（パスワード変更） | × | ○ |

### 権限マトリクス（ボード内操作）

ボードへのアクセスはオーナーのみに限定される。

| 操作 | 未認証 | 認証済み（非オーナー） | ボードオーナー |
|-----|--------|--------------------|--------------|
| タスクの作成・編集・閲覧 | × | × | ○ |
| タスクの削除 | × | × | ○ |
| ボードの削除 | × | × | ○ |

> 認証済みでも自分が作成したボード以外へアクセスした場合は HTTP 403 を返す。

> 認証済みユーザーが新規登録・ログイン画面へアクセスした場合はホーム画面へリダイレクトする。

> タスクの削除は確認ダイアログ等の対策は `docs/01-requirements/ユースケース.md` UC-013 および詳細設計書に委ねる。

## 非機能要件

学習目的のため高い可用性・スケーラビリティは求めないが、基本的な品質基準として以下を定義する。

| 項目 | 要件 |
|-----|------|
| パフォーマンス | API サーバーがリクエストを受信してからレスポンスを返すまでのサーバーサイド処理時間を 2 秒以内とする（同時接続 10 ユーザー以内の条件下。CSV エクスポートなどバッチ処理は除く） |
| 同時接続想定 | 学習用途のため最大 10 ユーザー同時接続を想定 |
| データ削除方式 | タスクの削除は物理削除とする（論理削除は対象外） |
| ログ | アプリケーションエラーログをサーバーサイドに出力する。保持期間・出力形式は詳細設計書で定義 |
| 可用性 | 学習目的のため SLA は定義しない。AWS EC2 単一構成で運用 |

## 制約条件

- 学習目的のため商用運用を前提としないが、セキュリティ対策は以下の区分で管理する
  - **本書で必須確定**: パスワードのハッシュ化（bcrypt）・CSRF 対策・SQL インジェクション対策・ログイン成功時のセッション ID 再生成（セッション固定攻撃対策）・HTTPS/TLS（セッション Cookie は Secure フラグ付きで発行。TLS 終端方式は詳細設計書で確定）
  - **実装必須・詳細設計書で方針決定**: XSS 対策・入力バリデーション
  - **スコープ外**: レート制限・IP ブロック・WAF
- フロントエンドは React + Vite による SPA 構成とする
- バックエンド API は REST（JSON）のみとする（GraphQL・WebSocket は対象外）
- 認証方式はセッション管理とする（OAuth・JWT・ソーシャルログインは対象外）
- セッションはサーバーサイドで PostgreSQL に永続化する。セッションには固定有効期限（絶対タイムアウト、想定: 24 時間）と無操作タイムアウト（アイドルタイムアウト、想定: 2 時間）の両方を設ける。先に到達した方を優先して失効とする。具体値は詳細設計書で確定
- インフラは Docker コンテナで構成し、AWS へデプロイする（確定: EC2 / [TBD]: データベースホスティング方式）
- 対応ブラウザはテスト実施時点の最新安定版（Chrome・Firefox・Edge）とする。N-1 バージョンは対象外（Safari は WebKit 固有の挙動差異への対応コストが学習スコープの費用対効果と合わないため対象外）
- モバイルネイティブアプリは対象外。レスポンシブ対応は 768px・1280px・1920px の3解像度を代表点として手動テストを行い、主要コンテンツの重なり・はみ出しがないことを確認する（手動テストチェックリストに含める）。768px 未満はテスト対象外とする

## 用語定義

本ドキュメントおよびプロジェクト全体で使用するドメイン用語の定義。

| 用語 | 定義 | 補足 |
|-----|------|------|
| ボード | タスクをまとめる管理単位 | Trello における「ボード」に相当 |
| タスク / カード | ボードに属する作業単位。本プロジェクトでは「タスク」で統一する | Trello における「カード」と同義 |
| ステータス | タスクの進捗状態を表す区分。未着手 / 進行中 / 完了 の固定3段階。カスタムステータスの追加・変更は現スコープ対象外 | カンバンのカラム区分にも対応 |
| ボードオーナー | ボードを作成したユーザー。ボード削除の権限を持つ | ボード単位のロール |
| サブタスク | 親タスクに紐付く子タスク。2 階層まで（親タスク → サブタスク）| サブタスクにサブタスクは持てない |
