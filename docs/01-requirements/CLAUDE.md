# 要件定義フェーズ - 作業ルール

## 1. インプット

- `docs/00-ideas/initial/` のアイデアメモを参照してください

## 2. アウトプットの粒度

- `要件定義成果物一覧.md` に記載された成果物に従って出力してください
- 1ファイルが肥大化し関心事が複数になった場合は、フォルダを作成して適切な粒度でファイルを分割してください
- 「何をするか」のみを記述してください。ユーザー・ビジネス視点で書いてください
- 概念データモデルはエンティティ・属性の洗い出し・関係の整理・必要情報の選定までとしてください
- 画面遷移図・モックアップはワイヤーフレームレベルまでとしてください

## 3. 書いてはいけないこと

以下は基本設計以降の領域であるため、要件定義では記述しないでください。

- 技術選定・技術スタック
- API のエンドポイント・HTTP メソッド・Request/Response 仕様
- 画面の詳細 UI（ボタン配置・入力項目の詳細・バリデーションルール）
- 認証・認可の実装方式
- エラーハンドリング方針
- インフラ構成・サーバー設計
- データモデルの物理設計（テーブル名・カラム名・型・制約）

## 4. Claude への判断指示

- 基本設計の領域に踏み込みそうな記述が必要になった場合は、書かずにユーザーに確認してください

## 成果物一覧

| 成果物 | 区分 | 目的 |
|-------|------|------|
| 要件定義書 | 主要文書 | プロジェクト全体の「約束書」です。発注者と開発者の合意内容を明文化し、後工程での判断基準となります。 |
| 業務フロー図（As-Is / To-Be） | 可視化 | 現状業務と将来あるべき姿を図示し、関係者全員が共通認識を持てるようにします。 |
| ユースケース図・一覧 | 機能整理 | 「誰が・何のためにシステムを使うか」を体系的に整理し、機能の抜け漏れとスコープを明確にします。 |
| 画面遷移図・モックアップ | UI合意 | ユーザー操作の流れを視覚化し、ワイヤーフレームレベルで認識合わせを行います。 |
| 概念データモデル（ER図） | データ整理 | システムが扱うエンティティ・属性・関係性を整理し、後続 DB 設計の土台とします。 |
| スコープ定義書 | 境界確定 | 「何をやるか」と「何をやらないか」を明確にし、スコープクリープを防ぎます。 |
| 用語集・ドメイン辞書 | 言語統一 | 業界用語・ドメイン用語を定義・統一し、ドキュメント全体の一貫性と認識齟齬を防ぎます。 |
