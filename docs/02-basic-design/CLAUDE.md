# 基本設計フェーズ - 作業ルール

## 1. インプット

- `docs/01-requirements/` の確定済み要件定義書を参照してください

## 2. アウトプットの粒度

- `基本設計成果物一覧.md` に記載された成果物に従って出力してください
- 1ファイルが肥大化し関心事が複数になった場合は、フォルダを作成して適切な粒度でファイルを分割してください
- 「どう実現するか」の方針レベル（論理レベル）で記述してください

### DB 設計の粒度
- 多対多の関係分解・キーの整理・第3正規形までの正規化を行ってください
- 物理名（英語カラム名）・型・制約・インデックスの定義は詳細設計で行ってください

### API 設計の粒度
- エンドポイント一覧（URL パス・HTTP メソッド）・各 API の責務・認証要否・レスポンスの概要まで定義してください
- Request/Response の完全な JSON スキーマ・バリデーションルール・エラーコードの詳細は詳細設計で行ってください

### 画面設計の粒度
- 詳細レイアウト・入力項目・ボタン配置まで定義してください

## 3. 書いてはいけないこと

以下は詳細設計以降の領域であるため、基本設計では記述しないでください。

| 書いてはいけない内容 | 詳細設計での対応先 |
|---|---|
| テーブルの物理設計（物理名・型・制約・インデックス詳細） | `テーブル定義書.md` |
| Request/Response の完全な JSON スキーマ・バリデーションルール・エラーコード詳細 | `API詳細設計書.md` |
| クラス・メソッドの詳細設計 | `バックエンド・クラス・モジュール設計書.md` |
| 処理フロー・擬似コード | `ビジネスロジック設計書.md` |
| CI/CD パイプラインの詳細設定 | `CI-CD設計書.md` |
| ログ出力の詳細設計 | `ログ設計書.md` |
| 環境変数の定義 | `環境変数定義書.md` |
| セキュリティの実装詳細 | `セキュリティ設計書.md` |
| 要件定義の内容の転記・繰り返し | 要件は参照するにとどめてください |

## 4. Claude への判断指示

- 詳細設計の領域に踏み込みそうな記述が必要になった場合は、書かずにユーザーに確認してください
- 要件定義で `[検討中]` / `[TBD]` の項目に依存する設計が必要な場合は、ユーザーに確認してください

## 成果物一覧

| 成果物 | 区分 | 作成が必要な理由 |
|-------|------|----------------|
| アーキテクチャ設計書 | 主要文書 | 要件定義で決めた「何をするか」を「どう実現するか」に翻訳した主要文書です。詳細設計・実装フェーズの作業指針となります。 |
| システム構成図 | インフラ設計 | サーバー・ネットワーク・ミドルウェアなどの全体構成を可視化します。インフラ調達・セキュリティ設計・障害対応計画の基盤となります。 |
| インフラ設計書 | インフラ設計 | インフラ構成の詳細（環境・スペック・構築手順）を定義します。システム構成図を補完し、実際の調達・構築作業の指針となります。 |
| 画面一覧定義書 | UI設計 | 全画面の詳細レイアウト・入力項目・ボタン配置を定義します。フロントエンド開発者の実装仕様書となります。 |
| データモデル設計書 | DB設計 | 要件定義で整理した概念データモデルを、正規化・関係分解を経た論理データモデルに落とし込みます。 |
| 論理ER図 | DB設計 | テーブル間のリレーション・カーディナリティを図示します。データモデル設計書と対で参照し、DB構造を視覚的に把握するために必要です。 |
| API設計書 | 連携設計 | エンドポイント一覧・各 API の責務・認証要否・レスポンス概要を定義します。フロントエンド・バックエンド間の設計の合意書となります。 |
| データフロー設計書 | フロー設計 | システム内のデータの流れ・変換・受け渡しを図示します。機能間の連携とデータの整合性を確保するために必要です。 |
| バッチ・非同期処理設計書 | 出力設計 | バッチ処理の実行タイミング・フロー、非同期処理の設計を定義します。業務オペレーションに直結する要素です。 |
| 認証・認可設計書 | セキュリティ設計 | 認証方式・セッション管理・権限モデルを定義します。セキュリティリスクを体系的に管理するために必要です。 |
| エラーハンドリング設計書 | 品質設計 | エラーの分類・ハンドリング方針・ユーザーへの表示方法を定義します。一貫した品質と運用性を確保するために必要です。 |
| 非機能要件定義書 | 品質設計 | パフォーマンス・セキュリティ・可用性・保守性など、機能以外の品質要件を定義します。 |
