# 認証・認可設計書

## 認証方式

サーバーサイドセッション管理（ASP.NET Core の Distributed Session / ISession）を採用する。  
クライアントはセッション ID を Cookie に保持し、リクエスト毎にサーバーへ送信する。  
サーバーはセッション ID でセッションストアを検索し、ログイン済みユーザーを識別する。

## セッション管理

| 項目 | 内容 |
|-----|------|
| セッション保存先 | サーバーサイド（インメモリ or Redis — 初期実装はインメモリ） |
| Cookie 名 | `greenello_session` |
| Cookie 属性 | `HttpOnly` / `Secure`（本番）/ `SameSite=Strict` |
| アイドルタイムアウト | 30 分（操作があるたびに延長） |
| 絶対タイムアウト | 7 日間（延長不可） |
| セッション破棄タイミング | ログアウト / パスワード変更 / タイムアウト |
| パスワード変更後の挙動 | 変更ユーザーのすべての既存セッションを無効化 |

## 権限設計

| ロール名 | 説明 | アクセス可能機能 |
|---------|------|--------------|
| 未認証ユーザー | ログインしていないユーザー | 新規登録・ログインのみ |
| 登録ユーザー | ログイン済みユーザー | ホーム画面・アカウント設定・自分のボードの全操作 |
| ボードオーナー | 対象ボードを作成したユーザー | タスク CRUD・ステータス変更・CSV エクスポート・担当者アサイン・ボード削除 |

> ボードオーナーは boards テーブルの `owner_user_id` で識別する。  
> 同一ユーザーが複数のボードをオーナーとして持つことができる。

## 認可チェック順序

各 API リクエストで以下の順序でチェックを行う。

1. **認証チェック**: セッションが有効か（未認証なら 401）
2. **アカウントロックチェック**: アカウントがロックされていないか（423）
3. **ボードアクセスチェック**: リクエストユーザーが対象ボードのオーナーか（boards.owner_user_id と一致するか）（403）
4. **オーナー専用操作チェック**: ボード削除など、オーナー確認が済んでいれば追加チェック不要

## ログイン失敗ロック

| 項目 | 内容 |
|-----|------|
| 失敗カウント | users.failed_login_count でカウント |
| ロック閾値 | 10 回連続失敗 |
| ロック期間 | 30 分（users.locked_until に解除日時を記録） |
| ロック解除 | 時間経過で自動解除（次回ログイン成功時に failed_login_count をリセット） |
| ユーザーへのメッセージ | 「アカウントがロックされています。30分後に再度お試しください。」 |
| ログイン失敗時のメッセージ | 「メールアドレスまたはパスワードが正しくありません。」（どちらが誤りかは明示しない） |

## パスワードポリシー

| 項目 | 内容 |
|-----|------|
| 最小文字数 | 8 文字 |
| 最大文字数 | 72 文字（bcrypt 入力長制約による） |
| 文字種 | 半角英字と数字をそれぞれ1文字以上含むこと |
| ハッシュ方式 | bcrypt（コストファクター: 12） |

## CSRF 対策

`SameSite=Strict` Cookie による制限を採用する。  
同一オリジン以外からのリクエストには Cookie が付与されないため、CSRF 攻撃を防ぐ。  
（学習目的のため CSRF Token 方式は対象外）

## 未認証時の挙動

- 認証が必要なエンドポイントへの未認証アクセス: `401 Unauthorized` を返す
- フロントエンドは 401 レスポンスを受け取った場合、ログイン画面にリダイレクトする
- セッション Cookie が期限切れ: サーバーはセッションを認識できず 401 を返す
