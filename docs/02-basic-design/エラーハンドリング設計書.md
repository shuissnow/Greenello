# エラーハンドリング設計書

## 基本方針

- バックエンドはすべてのエラーを統一された JSON 形式で返す
- HTTP ステータスコードでエラー種別を表現し、`error.code` で詳細を伝える
- ユーザーに表示するメッセージは `error.message` に格納する
- スタックトレースや内部情報はレスポンスに含めず、サーバーログにのみ記録する
- フロントエンドは HTTP ステータスコードを基にエラー処理を行い、必要に応じて `error.code` で分岐する

## エラー種別一覧

| エラーコード | HTTP ステータス | 発生条件 | ユーザー向けメッセージ例 |
|-----------|-------------|---------|-------------------|
| `UNAUTHORIZED` | 401 | 未認証・セッション切れ | 「ログインが必要です。」 |
| `ACCOUNT_LOCKED` | 423 | ログイン失敗10回によるアカウントロック | 「アカウントがロックされています。30分後に再度お試しください。」 |
| `FORBIDDEN` | 403 | ボード非メンバー・オーナー限定操作への一般メンバーアクセス | 「この操作を行う権限がありません。」 |
| `NOT_FOUND` | 404 | 存在しないリソースへのアクセス | 「リソースが見つかりませんでした。」 |
| `CONFLICT` | 409 | メールアドレス重複登録 | 「このメールアドレスは既に使用されています。」 |
| `VALIDATION_ERROR` | 422 | 入力バリデーション違反（タイトル空・文字数超過など） | フィールド単位のエラーメッセージ（下記参照） |
| `INVALID_CREDENTIALS` | 401 | ログイン時のメールアドレス・パスワード不一致 | 「メールアドレスまたはパスワードが正しくありません。」 |
| `EXPORT_NO_TASKS` | 400 | CSV エクスポート対象が 0 件 | 「エクスポート対象のタスクがありません。」 |
| `INTERNAL_ERROR` | 500 | 予期しないサーバー内部エラー | 「サーバーエラーが発生しました。時間をおいて再度お試しください。」 |

### VALIDATION_ERROR のレスポンス例

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力内容を確認してください。",
    "fields": {
      "title": "タイトルは必須です。",
      "category": "カテゴリは20文字以内で入力してください。"
    }
  }
}
```

## ログ設計

| ログレベル | 出力条件 |
|---------|---------|
| INFO | 正常なリクエスト処理完了（主要 API 操作） |
| WARN | バリデーションエラー・認証失敗・アカウントロック発生 |
| ERROR | 予期しない例外・DB 接続エラー・メール送信失敗（リトライ上限到達） |

- ログには `requestId`・`userId`（認証済みの場合）・`timestamp`・`path`・`statusCode` を含める
- パスワードや Cookie などの機密情報はログに出力しない

## フロントエンドのエラー表示方針

- **401 Unauthorized**: ログイン画面へリダイレクトする
- **403 Forbidden**: 「権限がありません」のトーストを表示する
- **422 Validation Error**: フォームの各フィールド直下にエラーメッセージを表示する
- **409 Conflict**: フォームのエラー表示エリアにメッセージを表示する
- **423 Account Locked**: ログインフォームのエラーエリアにメッセージを表示する
- **500 Internal Error**: 「サーバーエラーが発生しました」のトーストを表示する
- **ネットワークエラー**: 「ネットワークエラーが発生しました。接続を確認してください。」のトーストを表示する

## Drag & Drop のオプティミスティック更新とエラー処理

1. タスクカードのドロップ時に UI を即時更新（オプティミスティック更新）
2. API リクエストが失敗した場合は元のステータスにロールバック
3. 「ステータスの更新に失敗しました。」のトーストを表示する

## バックグラウンドジョブのエラー処理

- タスク単位で try-catch を行い、1 件の送信失敗が後続処理を中断しないよう制御する
- リトライはサービス層で指数バックオフを使わず、固定間隔（5分・30分・1時間後）でスケジュール
- 最終失敗はログ（ERROR レベル）に記録する
