# 非機能要件定義書

## パフォーマンス

- タスク一覧の初期表示は 2 秒以内に完了すること（1ボードあたりタスク 200 件以下・一般的なブロードバンド環境を前提とする）
- リスト表示はページネーションを採用し、1ページあたり 50 件を表示する
- カンバン表示は1カラムあたり初期 50 件を表示し、スクロール末尾到達時に追加 50 件ずつ読み込む（上限は100件/カラム）
- Drag & Drop 操作の UI 視覚的応答は 100ms 以内であること（アニメーション開始まで）
- Drag & Drop 操作のサーバーへのステータス保存は 500ms 以内であること
- API レスポンスタイムは通常時 500ms 以内を目標とする
- CSV エクスポートは 1,000 件以内のタスクであれば 5 秒以内に完了すること

## セキュリティ

- 認証にはサーバーサイドセッション管理を使用する
- セッション有効期限: アイドルタイムアウト 30 分・絶対タイムアウト 7 日間。期限切れ後は再ログインを要求する
- セッション Cookie には `HttpOnly`・`Secure`・`SameSite=Strict` 属性を設定する
- CSRF 対策として SameSite=Strict Cookie によるリクエスト制限を実施する（学習目的のため CSRF Token 方式は対象外）
- パスワードはハッシュ化（bcrypt）して保存する
- パスワードポリシー: 8文字以上72文字以内・半角英字と数字をそれぞれ1文字以上含むこと（72文字上限は bcrypt の入力長制約による）
- API エンドポイントは認証済みユーザーのみアクセス可能とする
- ボード参加者はそのボード内のリソースにアクセス可能とし、非参加者のアクセスを拒否する（ボード単位のアクセス制御）
- ボード内リソース（タスク・サブタスク等）の操作権限はボードオーナーのみとする
- ログイン連続失敗 10 回でアカウントを 30 分間一時ロックする
- HTTPS 通信を前提とする（AWS 環境でのデプロイ時に対応）
- SQL インジェクション対策: Entity Framework Core のパラメータ化クエリを使用し、生 SQL は原則不使用とする
- XSS 対策: React の自動エスケープを活用し、`dangerouslySetInnerHTML` は使用しない

## 可用性

- 学習目的のため、SLA は設定しない
- AWS 上で稼働し、単一リージョン構成を前提とする
- Docker コンテナで構成するため、ローカル環境でも同一構成で動作可能とする
- データバックアップの自動化は対象外とするが、PostgreSQL のデータボリュームは Docker named volume で永続化する

## 保守性

- フロントエンド・バックエンド・DB・インフラをそれぞれ独立したコンテナで管理する
- 環境変数で開発・本番環境を切り替え可能とする
- API は RESTful 設計に従い、将来的な機能追加がしやすい構造とする
- コードは Git で管理し、機能単位でブランチを切って開発する

## メール通知

- メール送信には外部メール送信サービス（SendGrid 等）を使用する
- 期限切れ検出バックグラウンドジョブは1日1回（08:00 JST を目安）実行する
- 同一タスクへの通知メールは1日1通を上限とし、重複送信しない
- メール送信失敗時は最大 3 回リトライする（初回失敗から 5 分後・30 分後・1 時間後）。最終的に失敗した場合はログに記録する（ユーザーへの再送は対象外）

## テスト

- フロントエンドのテストフレームワーク: Vitest + React Testing Library
- バックエンドのテストフレームワーク: xUnit（.NET 標準）
- カバレッジ目標: ビジネスロジック層（サービス層）は 80% 以上を目標とする
- API 統合テスト: xUnit + `WebApplicationFactory` を使用してコントローラーレベルのテストを実施する
- E2E テストは初期スコープ外とする

## 制約・前提

- フロントエンドは React + Vite による SPA 構成とする
- バックエンドは C# .NET Web API を使用する
- データベースは PostgreSQL を使用する
- インフラは Docker + AWS（学習目的のため最小構成）
- 商用運用・大規模トラフィックは対象外
