# アーキテクチャ設計書

## 構成方針

- フロントエンドとバックエンドを明確に分離した SPA + REST API 構成を採用する
- バックエンドは Clean Architecture に準じたレイヤー分離（Controller → Service → Repository → DB）を行い、拡張性・テスト容易性を確保する
- Docker コンテナで各コンポーネントを独立管理し、ローカル・AWS 本番で同一構成を維持する
- 認証はサーバーサイドセッションで管理し、Cookie で状態を維持する

## 技術選定

| レイヤー | 採用技術 | 選定理由 |
|---------|---------|---------|
| フロントエンド | React 18 + Vite + TypeScript | SPA 開発の実績が豊富。Vite による高速ビルド。型安全性による品質向上。学習目的として実務標準に近い構成 |
| 状態管理 | React Query (TanStack Query) | サーバー状態のキャッシュ・同期を簡潔に管理できる。REST API との相性が良い |
| Drag & Drop | dnd-kit | React 向けアクセシブルな DnD ライブラリ。モダンで軽量 |
| UIスタイル | Tailwind CSS | ユーティリティファーストで開発速度が高い |
| バックエンド | C# .NET 8 Web API | 型安全・高パフォーマンス。Entity Framework Core との親和性。学習目的として採用 |
| ORM | Entity Framework Core | PostgreSQL との統合が容易。LINQ によるクエリ記述でSQLインジェクションを防ぎやすい |
| データベース | PostgreSQL 16 | オープンソースで実績豊富。EF Core との相性良好 |
| 認証 | サーバーサイドセッション (ASP.NET Core Session) | JWT と比較してセッション無効化が容易。要件どおりのアイドルタイムアウト・絶対タイムアウトを実装しやすい |
| リバースプロキシ | Nginx | 静的ファイル配信・API プロキシを1台で担当できる軽量な構成 |
| コンテナ | Docker + Docker Compose | 環境再現性の確保。開発・本番の差異を最小化 |
| インフラ | AWS (EC2 + RDS or EC2 + Docker) | 学習目的での最小構成デプロイ |
| メール | SendGrid | 無料枠あり・SMTP/HTTP API どちらも使いやすい外部サービス |
| フロントテスト | Vitest + React Testing Library | Vite と統合しやすいテストランナー。コンポーネントのユーザー視点テストに適する |
| バックテスト | xUnit + WebApplicationFactory | .NET 標準テストフレームワーク。統合テストが書きやすい |
| API ドキュメント | OpenAPI 3.1 + Scalar | OpenAPI でスキーマを一元管理し、Scalar UI（`GET /scalar/v1`）で開発中の API 仕様を閲覧可能にする |

## バックエンド レイヤー構成

```
Controllers     HTTP リクエスト受付・レスポンス返却
    │
Services        ビジネスロジック（テスト対象の中心）
    │
Repositories    DB アクセス抽象化（EF Core DbContext）
    │
Database        PostgreSQL
```

## 設計上の制約・前提

- 商用運用・大規模トラフィックは対象外のため、シンプルさを優先する
- CSRF 対策は `SameSite=Strict` Cookie のみとし、CSRF Token は導入しない
- E2E テストは初期スコープ外とする
- カスタムステータスは対象外（Inbox / 進行中 / 完了 の3種固定）
- 親子タスクは2階層のみ（サブタスクへのサブタスク追加は UI レベルで不可）
